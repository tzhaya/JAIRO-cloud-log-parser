# 追加でまとめておくことを推奨する情報

## 1. WEKO3のバージョンとソース確認時点

統計ロジックは、WEKO3のバージョンや改修によって変更される可能性があります。

次を記録しておくことを推奨します。

- 確認したGitHubリポジトリ
- 確認したコミットID
- 確認日
- 利用中のJAIRO Cloud環境のバージョン
- 将来再確認すべき主要ファイル

これにより、仕様変更時に差分を追跡できます。

## 2. 用語集

利用者向けに、次の用語を整理した短い用語集があると有用です。

- アイテム
- ファイル
- アイテム公開状態
- アイテムのアクセス権
- `accessrole`
- `open_access`
- `open_date`
- 非ログイン利用
- サイトライセンス
- OAダウンロード

特に、「アイテムのアクセス権」と「ファイルのaccessrole」は別であることを強調する必要があります。

## 3. 問い合わせ対応用チェックリスト

数値差異の問い合わせに対して、次の順で確認するチェックリストを作成すると対応が安定します。

1. 対象年月を確認
2. ファイル名を確認
3. 登録インデックス名を確認
4. 全体側の件数を確認
5. OA側の件数を確認
6. JSONの `item_files[].accessrole` を確認
7. `open_date` の場合は日付を確認
8. アイテムのアクセス権と混同していないか確認
9. 旧データ移行の可能性を確認
10. Power Queryの結合キーを確認

## 4. 集計値の名称方針

Excelや説明資料では、次の名称を推奨します。

| 現在の名称 | 推奨名称 |
|---|---|
| No. Of File Downloads | 全ファイルダウンロード |
| Non-Logged In User | 非ログイン利用 |
| Open-Access No. Of File Downloads | WEKO3 OA設定ファイルのダウンロード |
| FileDownload (OA) | FileDownload (`accessrole=open_access`) |

OAという語だけでは、実際の公開状態との違いが伝わりにくいためです。

## 5. 実質的な公開ファイル利用数を把握する方法

`Open-Access No. Of File Downloads` だけでは、現在誰でも利用できる全ファイルを把握できません。

より正確に把握するには、次の追加処理が必要です。

- 各ファイルの `accessrole` を取得
- `open_access` を対象にする
- `open_date` について公開日経過を判定する
- 必要に応じてアイテム公開状態を確認する
- 統計イベントとファイルメタデータを結合する

これはWEKO3標準レポートだけでは完結しないため、別ツールまたは追加データ取得が必要です。

## 6. データ移行時の設定差

WEKO2等から移行したデータでは、現在のWEKO3で新規登録したデータと異なる `accessrole` が設定されている可能性があります。

次を調査対象として整理しておくと有用です。

- 移行データで `open_date` が多いか
- `accessrole` 空欄が存在するか
- アイテムアクセス権とファイルアクセス設定が一致しているか
- 一括修正が必要か
- 修正によって過去統計が変わるか

## 7. 再現可能な検証データ

今後の改善や不具合確認のため、匿名化した小規模サンプルを保存することを推奨します。

最低限、次のケースを含めます。

- `open_access`
- 公開日前の `open_date`
- 公開日後の `open_date`
- `open_login`
- `open_no`
- `accessrole` 空欄
- 非ログイン利用
- ログイン利用
- サイトライセンス利用

## 8. 解釈上の限界

レポートの数値には、次の限界があります。

- 非ログイン利用者が学外者とは限らない
- OA統計が実質的な全公開ファイルを表さない
- 内訳列は単純加算できない
- ファイル名だけでは一意でない場合がある
- 短時間重複除外によりWebアクセスログと一致しない
- 過去の設定変更が過去イベントに遡及反映されるとは限らない

これらを「既知の制約」としてREADMEまたはFAQにまとめると、ツール利用者の誤解を減らせます。
