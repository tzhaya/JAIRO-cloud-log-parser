# AGENTS.md

このファイルは、このリポジトリで作業する AI コーディングエージェント向けのガイドです。

## リポジトリの概要

このリポジトリは、JAIRO Cloud（WEKO3）から出力された月次の定型レポートを結合する、単一成果物の Excel ツールを管理しています。

- 成果物は `JAIRO_Cloud_log_parser_YYYYMMDD.xlsx` です。
- データの取得・変換ロジックは、ワークブック内の Power Query（M 言語）に埋め込まれています。
- 通常のソースコード、ビルドシステム、自動テストはありません。
- `README.md` は日本語のエンドユーザー向けドキュメントです。
- ライセンスは CC0 1.0 です。
- リリース時は、日付サフィックス付きの新しいワークブックをコミットします（例：`JAIRO_Cloud_log_parser_20250104.xlsx`）。

## ワークブックの取り扱い

`.xlsx` はバイナリファイルのため、通常の Git 差分では変更内容を確認できません。

- Power Query の機能変更は Excel の Power Query エディターで行う必要があります。
- リポジトリ上で `.xlsx` を直接編集して M コードを変更することはできません。
- 既存のワークブックを上書きするのではなく、リリース規則に従って日付サフィックス付きの新しいファイルを作成してください。
- バイナリ変更を行った場合は、変更したクエリ、シート、確認手順および確認結果を明示してください。

### Excel を使わずに M コードを閲覧する方法

`.xlsx` は ZIP アーカイブです。Power Query の定義は、対応する `itemProps*.xml` に `DataMashup` と記載された `customXml/item9.xml` 内に、base64 エンコードされた DataMashup バイナリとして格納されています。

1. base64 データをデコードします。
2. 先頭 8 バイトのヘッダーを読みます。バイト 4〜7 は、内側の ZIP の長さを表すリトルエンディアン値です。
3. ヘッダーを除いたデータを ZIP として展開します。
4. `Formulas/Section1.m` を読みます。

この手順は閲覧・調査用です。機能変更は Excel の Power Query エディターで行ってください。

## Power Query のデータフロー

### 入力

ユーザーは「Usage and Config」シートのセル B9に、定型レポートを格納したフォルダのパスを貼り付けます。この値は、テーブル `Path` の列 `定型レポートの場所` として参照されます。

すべてのクエリはこのパスを対象とする `Folder.Files` から始まり、ファイル名のプレフィックスで対象ファイルを絞り込みます。

- `logReport_FileDownload_{YYYY}-{MM}.tsv`
  - 2 つのセクションが縦に連結されています。
  - 前半は `No. Of File Downloads`、後半は OA セクションです。
  - 両者の間にマーカー行 `Open-Access No. Of File Downloads` があります。
  - `f_FileDownload` はマーカーより前、`f_FileDownloadOpenaccess` はマーカーより後の行を抽出します。
- `logReport_DetailView_{YYYY}-{MM}.tsv`
  - 4 列からなる単一セクションです。

### TSV の前提

- 文字コード：UTF-8（コードページ 65001）
- 区切り文字：タブ
- 先頭 4 行：前置き行としてスキップ
- 5 行目：ヘッダーとして昇格

### ファイルごとの変換

変換関数 `f_FileDownload`、`f_FileDownloadOpenaccess`、`f_DetailView` を、条件に一致するすべてのファイルへ適用します。

`date` 列（例：`2024-12`）は、ファイル名に対する固定オフセットの `Text.Middle` で生成されます。

- FileDownload 系：オフセット 23
- DetailView 系：オフセット 21

ファイル名の形式が変わると、この処理は壊れます。ファイル名に関する変更では、抽出位置への影響を必ず確認してください。

### 出力クエリとシート

- `FileDownload`：全月のダウンロードデータを行レベルで連結
- `Filedownload (OA)`：全月の OA ダウンロードデータを行レベルで連結
- `DetailVIew`：全月の詳細閲覧データを行レベルで連結
- `FileDownload by Month`：ダウンロードデータを月ごとに `List.Sum` で集計
- `DetailVIew by Month`：詳細閲覧データを月ごとに `List.Sum` で集計

`DetailVIew` は大文字・小文字の誤りを含む実在のクエリ名です。参照時は表記を変更しないでください。

### サンプルファイル依存

各 `f_*` 関数には、列名の導出に使うサンプルバイナリをまとめる `サンプル`／`パラメーター` クエリの組があります。これは Power Query の「サンプル ファイル」パターンです。

サンプルクエリは、フォルダ内で最初に一致したファイルを `{0}` で選択します。そのため、次の場合は関連するすべてのクエリが失敗します。

- 対象フォルダが空の場合
- 想定した名前のファイルが存在しない場合
- 最初の一致ファイルの内容や形式が想定と異なる場合

## 変更時のルール

- 統計値の意味を推測で変更しないでください。`docs/` の調査結果と WEKO3 の仕様を根拠にしてください。
- 全体ダウンロード数と OA ダウンロード数を合算しないでください。
- 利用者区分の各列を、排他的な分類とみなして単純合計しないでください。
- Power Query のステップ名は日本語版 Excel による日本語名です。Power Query エディターで変更する場合も、この命名慣例を維持してください。
- 実在するクエリ名やシート名の誤記を、影響調査なしに修正しないでください。名前の変更は参照関係を壊す可能性があります。
- README や利用者向け説明では、「全体」「OA」「利用者区分」の関係を曖昧にしないでください。
- 変更範囲は対象 Issue と実装計画に合わせ、無関係な変更を混在させないでください。

## 検証

自動テストはありません。変更内容に応じて、少なくとも次を確認してください。

- 対象の月次 TSV を指定したとき、すべてのクエリをエラーなく更新できること。
- FileDownload の通常セクションと OA セクションが、マーカー行の前後で正しく分離されること。
- DetailView が想定どおり 4 列で読み込まれること。
- ファイル名から `date` 列が正しく生成されること。
- 複数月のファイルが行レベルで連結され、月別クエリでは正しく集計されること。
- 空フォルダ、不正なファイル名、不正な TSV 形式など、変更対象に関係する異常系の挙動を確認すること。
- 統計値の説明を変更した場合、`docs/` の仕様と矛盾していないこと。

検証できない項目がある場合は、未検証であることと、その理由を作業報告に明記してください。
